Getting Started

Dain

A low-level, general-purpose language designed for explicit control flow, zero-cost abstractions, and a clear compilation pipeline.

v0.1.0-dev — early demo

What is Dain?

Dain is a compiled, statically-typed language with an optional interpreted mode for fast development cycles. It exposes the full compilation pipeline as distinct CLI commands — so you always know exactly what stage your code is at.

The Pipeline

Each CLI command corresponds to a stage. Every stage includes all previous stages:

check
errors only
→
compile
IR / object
→
build
native binary
→
execute
run existing
→
run is a shortcut for build + execute. Add --interpreted to any of these commands to use the bytecode path instead of native compilation.

Quick Start

Create a file main.dain and write your first program:

dain 
import std.io;

fn main() {
    println("Hello World!");
}

Then run it:

shell 
dain run main.dain
Hello World!

Full Quick Start guide →

Commands

Parse and type-check. Report errors, produce no output.
compile
Compile to IR or object file. No linking.
build
Compile and link to a native binary.
execute
Run an existing binary. Fails if not found.
run
Build then execute in one step.

Installation

Install Dain with a single command. The installer handles everything — the ABI, the latest compiler binary, and all required dynamic libraries.

Linux & macOS

Run the install script from your terminal:

shell 
curl -sSL https://raw.githubusercontent.com/GalaxyHaze/Dain/main/install.sh | sh

Windows

Run the PowerShell installer from an elevated terminal:

powershell 
irm https://raw.githubusercontent.com/GalaxyHaze/Dain/main/install.ps1 | iex
→
The installer automatically fetches the latest compiler binary, the ABI layer, and all required dynamic libraries. No manual dependency management needed.

Verify

shell 
dain version
Dain Programming Language
Version: 0.1.0-dev
docs / quick start
Getting Started

Quick Start

Write and run your first Dain program in under a minute.

1. Create a source file

Create a file called main.dain with the following content:

dain 
import std.io;

fn main() {
    println("Hello World!");
}

2. Run it

Use dain run to compile and execute in one step:

shell 
dain run main.dain
Hello World!

3. Or iterate faster with interpreted mode

Skip native compilation entirely during development — the bytecode path is faster to invoke:

shell 
dain run --interpreted main.dain
Hello World!

4. Check for errors without building

Hook dain check into your editor's save event for instant feedback:

shell 
dain check main.dain
[ok] Check passed: main.dain

5. Build a release binary

shell 
dain build main.dain -m release -o bin/hello
./bin/hello
Hello World!
→
Ready to grow your project? Add a DainProject.toml at the root and run dain with no arguments to build automatically.

Interpreted Mode

Every command that produces or consumes a compiled artifact accepts --interpreted. This switches the target from native binary to Dain Bytecode (.nbc).

NativeInterpreted equivalent
dain compile main.daindain compile --interpreted main.dain
dain executedain execute --interpreted
dain rundain run --interpreted
i
build always produces a native binary. There is no build --interpreted.

Project Resolution

Commands that accept an optional input file will fall back to reading DainProject.toml in the current directory when no file is specified. If neither is found, the command fails with an explicit error.

docs / cli / check
CLI Reference

dain check

Parse and type-check a source file. Reports all errors to stderr. Produces no output files.

Synopsis

shell
dain check [input] [-m MODE] [-v]

Arguments

ArgumentDescription
inputSource file to check. Optional — reads DainProject.toml if omitted.

Examples

shell
# check a specific file
dain check main.dain

# check with verbose output
dain check main.dain -v

# check entry from DainProject.toml
dain check

Output

On success, prints [ok] Check passed: <file>. On failure, all errors are printed to stderr and the exit code is non-zero. No files are written to disk.

→
Use check in your editor's save hook or CI pipeline for fast feedback without touching the build artifacts.
docs / cli / compile
CLI Reference

dain compile

Compile a source file to an IR/object file (native) or bytecode file (interpreted). Does not link.

Synopsis

shell
dain compile <input> [-o OUTPUT] [--interpreted] [-m MODE] [-I DIR] [-v]

Arguments & Flags

FlagDescription
inputSource file (.dain). Required.
-o, --outputOutput file. Defaults to a.o (native) or a.nbc (interpreted).
--interpretedCompile to Dain Bytecode (.nbc) instead of native object.
-m, --modeBuild mode. See Global Flags.
-I, --includeAdd include directory. Repeatable.

Examples

shell
# compile to native object
dain compile main.dain -o main.o

# compile to bytecode
dain compile --interpreted main.dain -o main.nbc

# release mode with include paths
dain compile main.dain -m release -I ./libs -I ./vendor
docs / cli / build
CLI Reference

dain build

Compile and link to a native binary. Always produces a native executable — there is no interpreted mode for build.

Synopsis

shell
dain build [input] [-o OUTPUT] [-m MODE] [-I DIR] [-v]

Project Resolution

When input is omitted, build reads DainProject.toml from the current directory. CLI flags take precedence over toml values.

i
The -m flag only overrides the toml mode if you explicitly pass it. The default debug does not override a toml that specifies release.

Examples

shell
# build from DainProject.toml
dain build

# build a specific file
dain build main.dain -o bin/app

# release build
dain build -m release -o bin/app
docs / cli / execute
CLI Reference

dain execute

Run an existing binary or bytecode file. Does not compile anything. Fails with an explicit error if the target is not found.

Synopsis

shell
dain execute [target] [--interpreted] [-v]

Target Resolution

When target is omitted, reads output from DainProject.toml. If no toml exists, the command fails immediately.

InvocationResolves to
dain executetoml.output
dain execute --interpretedtoml.output.nbc
dain execute bin/appbin/app

Examples

shell
# run binary from toml
dain execute

# run specific binary
dain execute bin/app

# run bytecode
dain execute --interpreted bin/app.nbc
!
execute never compiles. If the binary doesn't exist, it tells you exactly which command to run first.
docs / cli / run
CLI Reference

dain run

Build and execute in a single step. Equivalent to dain build && dain execute.

Synopsis

shell
dain run [input] [-o OUTPUT] [--interpreted] [-m MODE] [-I DIR] [-v]

Interpreted Mode

With --interpreted, run calls compile --interpreted followed by execute --interpreted. No linking occurs.

CommandEquivalent to
dain rundain build && dain execute
dain run --interpreteddain compile --interpreted && dain execute --interpreted

Examples

shell
# build and run from toml
dain run

# run a specific file
dain run main.dain

# interpreted mode (fast iteration)
dain run --interpreted main.dain

# release build and run
dain run -m release
docs / cli / global flags
CLI Reference

Global Flags

These flags are available on all commands that produce or consume compilation artifacts.

  • -m, --mode <MODE> Build mode. One of debug, dev, release, fast, test. Default: debug
  • -o, --output <FILE> Output file path. Defaults vary by command.
  • -I, --include <DIR> Add a directory to the import search path. Can be repeated.
  • --interpreted Use bytecode compilation path instead of native. Not available on build.
  • --emit <FORMAT> Emit an intermediate representation: ast, ir, asm, obj, bin. Not yet implemented.
  • --target <TRIPLE> Cross-compilation target triple (e.g., x86_64-linux-gnu). Not yet implemented.
  • -v, --verbose Print detailed progress information.

Build Modes

ModeDescription
debugFull debug info, no optimizations. Default.
devLight optimizations, debug info retained.
releaseFull optimizations, no debug info.
fastAggressive optimizations, may sacrifice safety checks.
testEnables test harness and assertions.
docs / DainProject.toml
Project

DainProject.toml

The project manifest file. Placed at the root of your project, it lets build, run, check, and execute work without explicit file arguments.

Example

toml
# DainProject.toml

[project]
name    = "my_project"
version = "0.1.0"
entry   = "src/main.dain"

[build]
mode    = "debug"
output  = "bin/my_project"
include = ["libs", "vendor"]

Fields

[project]

FieldTypeDescription
namestringProject name.
versionstringSemantic version string.
entrystringPath to the main source file.

[build]

FieldTypeDescription
modestringDefault build mode. Overridden by -m.
outputstringOutput binary path.
includestring[]Include directories, equivalent to repeated -I.

Precedence

CLI flags always take precedence over toml values. The toml is only read when a value is not provided on the command line.

→
Running dain with no arguments in a directory that has a DainProject.toml will automatically invoke dain build.

Roadmap

Roadmap

Dain is at v0.1.0-dev. The language and toolchain are under active development.

!
APIs and language syntax may change before v1.0. Do not use in production.

Currently Available (v0.1.0)

Core CLI pipeline: check, compile, build, execute, run. Native compilation and interpreted bytecode mode. DainProject.toml project manifest.

Planned Features

Cross-compilation via --target. IR/ASM/obj emission via --emit. Full standard library documentation. Package manager. LSP server.
